Getting Started

Build applications that extend the ICHIN OS cognitive ecosystem.

Quick Start

Apps in ICHIN OS are sandboxed, AI-aware extensions that integrate into workspaces, the Orb notification system, and the Spotlight interface.

# Create a new app
npx create-ichin-app my-app --template typescript

# Install dependencies
cd my-app && npm install

# Start development
npm run dev

Prerequisites

  • Node.js 18+ or Python 3.11+
  • ICHIN OS Developer SDK
  • Access to the App Store Developer Portal

App Structure

Every ICHIN app requires a manifest.json file that declares identity, permissions, AI compatibility, and workspace integration.

App Architecture

ICHIN OS supports three app types, each with different integration levels.

Native Ichin Apps

Built specifically for ICHIN OS. Full AI, memory, and workspace integration. Best performance.

Web Apps (Wrapped)

Run inside a sandbox container. Limited system access. UI integrates into workspace system.

External Apps (Compatibility Mode)

Linux / Windows / Android apps executed via compatibility layers. Restricted AI integration.

Sandbox Architecture

All apps run in isolated containers with no direct OS access. Communication passes through the Orchestrator Service.

App → Sandbox → Permission Check → API Gateway → Orchestrator → System

AI API

Query the ICHIN AI system from your app via the Orchestrator Service (port 8011).

Orchestrate (main AI endpoint)

POST http://localhost:8011/orchestrate
{
  "request": "Summarize my recent notes",
  "workspace": "study",
  "mode": "normal"
}

Classify Request

POST http://localhost:8011/classify
{
  "request": "Plan my study schedule",
  "workspace": "study"
}

Get Session Context

GET http://localhost:8011/context/{session_id}

All AI API calls are routed through the Orchestrator for permission checking and resource management. The Orchestrator selects agents, runs weighted voting, and validates decisions.

Memory API

Read and write structured data to the ICHIN Memory Engine (port 8013).

Store Memory

POST http://localhost:8013/memory/store
{
  "key": "user_preference_theme",
  "value": "dark",
  "type": "fact",
  "workspace": "coding",
  "importance": 0.8
}

Query Memory

POST http://localhost:8013/memory/query
{
  "query": "coding preferences",
  "workspace": "coding",
  "limit": 5
}

Promote to Long-Term

POST http://localhost:8013/memory/promote
{
  "memory_id": "mem_123",
  "target_layer": "long_term"
}

Memory access is restricted by app permissions. 4 layers: ephemeral, working, long-term semantic, structured. Memory decays over 30-day half-life unless reused. Workspace isolation enforced.

Calendar API

Calendar operations are handled by the Calendar Agent within the Agents Service (port 8012).

Execute Calendar Agent

POST http://localhost:8012/agent/execute
{
  "agent": "calendar",
  "request": "Create a study session tomorrow at 10am for 2 hours",
  "workspace": "study",
  "mode": "normal"
}

The Calendar Agent detects events, proposes entries, resolves conflicts, and optimizes schedules. Requires CALENDAR_ACCESS permission. Events are linked to workspaces for contextual scheduling.

File API

File operations are managed by the App Runtime Service (port 8015).

Install App (grants sandbox)

POST http://localhost:8015/app/install
{
  "manifest": {
    "name": "My App",
    "version": "1.0.0",
    "app_type": "native",
    "permissions": [
      { "type": "file_access", "granted": true, "user_approved": true }
    ]
  }
}

Run App

POST http://localhost:8015/app/{id}/run

All app file operations are sandboxed. No direct OS access. Requires FILE_ACCESS permission. Files are automatically indexed by the AI system via the memory engine.

Notification API (Orb System)

Trigger AI Orb events through the UI System Service (port 8014).

Trigger Orb Event

POST http://localhost:8014/orb/notify
{
  "message": "Document analysis complete",
  "state": "active",
  "source": "my-app-name"
}

Orb State

GET http://localhost:8014/orb/state

Orb states: idle (faint glow), active (30s countdown window), critical (strong glow, requires action). Requires NOTIFICATIONS_ACCESS permission. Notifications respect focus mode rules — suppressed in deep_focus, blocked in lock mode.

Permission Model

Every app must declare its permissions. No permission = no access.

Permission Types

AI_ACCESS
Use the ICHIN AI system
MEMORY_READ
Read user memory (restricted)
FILE_ACCESS
File system access
NETWORK_ACCESS
Internet usage
WORKSPACE_INTEGRATION
UI embedding in workspaces
CALENDAR_ACCESS
Calendar operations
NOTIFICATIONS_ACCESS
Trigger Orb notifications

Rules

  • Permissions must be user-approved
  • Sensitive permissions require explicit confirmation
  • AI can suggest but not override permissions
  • Users can revoke permissions at any time

TypeScript SDK

Build ICHIN apps with TypeScript for full type safety. Packages are in ichin-os/packages/.

# In your monorepo:
npm install @ichin/shared-types   # shared type definitions
npm install @ichin/ai-sdk         # AI agent & workflow builders
npm install @ichin/ui-components  # React component library
npm install @ichin/permissions-model  # permission utilities
import { AgentBuilder } from '@ichin/ai-sdk'
import { AIOrchestratorClient } from '@ichin/ai-sdk'
import type { WorkspaceType, FocusMode } from '@ichin/shared-types'
 
 const client = new AIOrchestratorClient('http://localhost:8011')
 
 // Query AI via Orchestrator
 const result = await client.orchestrate({
   request: 'Summarize my day',
   workspace: 'study',
   mode: 'normal',
 })
 
 // Build a custom agent
 const agent = new AgentBuilder()
   .setName('My Assistant')
   .setRole('Helps with daily tasks')
   .setReasoningStyle('structured')
   .setTools(['calendar', 'notes'])
   .build()

Python SDK

Build ICHIN AI agents and automation tools with Python. See the app template at ichin-os/ecosystem/app-templates/python-agent/.

# Agents integrate via HTTP to ICHIN services:
from agent import StudyAssistantAgent
import asyncio
 
 agent = StudyAssistantAgent()
 asyncio.run(agent.on_activate({"workspace": "study"}))
 
 # Uses Orchestrator (port 8011): POST /orchestrate
 result = await agent.process_task({
     "action": "summarize",
     "content": "Biology chapter 5 notes",
 })
 
 # Uses Memory Engine (port 8013): POST /memory/store
 await agent.write_memory("last_topic", "Biology")
 
 # Uses UI System (port 8014): POST /orb/notify
 await agent.notify("Summary ready", "active")

App Store Publishing

Submit your app to the ICHIN App Store for distribution.

Submission Process

  1. Prepare your app with a valid manifest.json
  2. Submit through the App Store API (POST http://localhost:8021/api/store/apps)
  3. AI simulation review checks security, sandbox, performance, and AI compatibility
  4. App receives a review score and is published if approved

AI Review Process

Every app undergoes automated AI simulation testing:

  • Security Validation — Scans permissions and identifies risks
  • Sandbox Compliance — Ensures proper isolation
  • Performance Test — Checks resource usage patterns
  • AI Compatibility Test — Validates AI integration quality

Categories

Apps can be published under: Productivity, Study Tools, Coding Tools, AI Agents, Learning Modules, System Extensions.

AI Integration Guide

Make your app AI-aware and contextually intelligent.

AI App Types

Assistant Apps

Context-aware helpers that run inside workspaces. They provide inline assistance based on the user's current activity.

Agent Apps

Custom AI agents that integrate into the Council system. They can participate in multi-agent reasoning for complex tasks.

Automation Apps

Trigger workflows and connect apps together. They define event-driven or time-based automation sequences.

Integration Rules

  • AI apps must use the Orchestrator for execution
  • All AI operations must follow memory restrictions
  • Apps must declare their AI behavior scope in the manifest
  • AI system always has priority over app execution

Best Practices

Follow these guidelines to build high-quality ICHIN apps.

Design Principles

  • Apps are extensions of cognitive workflows, not standalone experiences
  • Visual design must integrate into the workspace system — no foreign UI styles
  • Use the Orb system for notifications, not custom toast UIs
  • Respect Focus modes — heavy operations should be throttled

Performance

  • Apps must not block the AI system
  • Resource limits are enforced per app
  • Use memory efficiently — apps share resources with the rest of ICHIN

Security

  • Sandbox isolation is mandatory — no direct OS access
  • API communication is encrypted
  • The AI security agent monitors app behavior
  • Rollback system available for system-level changes
  • Request minimum permissions — no permission = no access

UX Integration

  • Integrate into workspace system (Service 4)
  • Support Spotlight for quick actions
  • Use Orb for attention requests
  • Follow the ICHIN design language
